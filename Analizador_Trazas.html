Analizador_Trazas.html

Documento base para la v1 del analizador de trazas de firma. Resume lo acordado hasta ahora para poder continuar más adelante con una versión publicada en GitHub Pages y, si se desea, evolucionar después a una herramienta con lógica de análisis.

Objetivo v1

Construir una herramienta básica, prudente y útil. La primera versión no debe intentar resolver todos los casos, sino solo los que están bien acotados y se pueden diagnosticar con reglas claras.

  • Fallo pasarela / SAML.
  • Fallo Portafib / sesión.
  • Fallo formulario / pre-firma.
  • Más adelante: entorno Autofirma, Cl@ve, FIRE y registro.
Entrada prevista
  • Área de texto grande para pegar trazas de SistraHelp.
  • Checks de contexto: certificado local / Cl@ve / no informado.
  • SO: Windows, Mac, iPad, iPhone, Android, no informado.
  • Checks adicionales: error SAML, página en blanco, traza en orden reciente → antiguo.
Salidas deseadas
  • Diagnóstico técnico.
  • Texto corto para pegar en el ticket.
  • Correo sugerido listo para copiar.
  • Nivel de confianza: alto, medio o bajo.

Reglas funcionales acordadas

1. Fallo pasarela / SAML

Si el ciudadano indica que la página queda en blanco o ve un error SAML, el problema puede estar en la pasarela de acceso o autenticación. En estos casos puede no existir traza útil en SistraHelp.

  • No usar SistraHelp como única prueba.
  • Conviene añadir un check específico para SAML.
  • Ese check puede desplegar ayuda con ejemplos de mensajes SAML y su significado.

2. Portafib / sesión

Portafib queda bien delimitado con una regla prudente:

  • No aparece TR_SGI.
  • Aparecen errores de flujo como Fluxe no vàlid o Excepció sessió firma.

Esto apunta a [Error Acceso-Sesión] – Portafib/Soffid pre-proveedor.

3. Formulario / pre-firma

Si no aparece TR_SGI y tampoco aparecen los errores de flujo anteriores, el caso debe clasificarse como fallo de formulario o incidencia previa al inicio de firma.

  • No atribuirlo a Cl@ve, FIRE o Autofirma.
  • No sobreinterpretar una traza incompleta.

Matices importantes

  • Las trazas suelen venir en orden más reciente → más antiguo. La futura lógica debe tenerlo en cuenta.
  • El error 227 no debe usarse como evidencia fuerte de Portafib. Se considera genérico y puede deberse a saturación puntual o a un fallo recuperable recargando.
  • El mensaje que ve el ciudadano de tipo Cargando el asistente también puede ser transitorio y no debe forzar un diagnóstico definitivo.
  • En esta v1 se dejan fuera otros literales hasta confirmar mejor su utilidad diagnóstica.

Interfaz propuesta

Bloque 1: Entrada - Textarea para trazas - Check: Certificado local - Check: Cl@ve - Selector de SO - Check: El ciudadano ve error SAML - Check: La página queda en blanco - Check por defecto: Traza en orden reciente → antiguo Bloque 2: Resultado - Eventos detectados - Regla aplicada - Diagnóstico - Nivel de confianza Bloque 3: Acciones - Texto breve para ticket - Correo sugerido - Botón copiar

Ejemplo de estructura de reglas

{ id: "portafib_preproveedor", condiciones: [ "no aparece TR_SGI", "aparece Fluxe no vàlid o Excepció sessió firma" ], diagnostico: "[Error Acceso-Sesión] – Portafib/Soffid pre-proveedor", texto_ticket: "No consta inicio de firma y sí errores de flujo previos al proveedor.", correo_id: "correo_portafib", nivel_confianza: "alto" }

La idea es separar la lógica del diagnóstico de las plantillas de correo, para poder cambiar los correos sin tocar las reglas.

Próximo paso recomendado

Antes de programar la versión interactiva, conviene cerrar una tabla de reglas v1 con:

  • Nombre de la regla.
  • Condiciones exactas.
  • Diagnóstico resultante.
  • Texto corto de ticket.
  • Plantilla de correo asociada.
  • Nivel de confianza.

Después de eso, la siguiente iteración natural sería convertir este documento en una página funcional con análisis en JavaScript y publicación en GitHub Pages.
