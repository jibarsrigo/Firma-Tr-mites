Pantalla en blanco al iniciar trámite (Acceso‑Acceso)

Escenario: Al pulsar «Iniciar trámite» en la Sede Electrónica CAIB, la página queda en blanco y no se carga la pasarela de identificación de Cl@ve, tanto usando certificado local como Cl@ve, o el ciudadano visualiza un error SAML o un error de conexión/timeout contra la pasarela.

Trazas:

En la mayoría de fallos generales de la pasarela Cl@ve, el formulario no llega a iniciarse y NO aparece TR_FRI ni trazas en SistraHelp.
❌ NO aparece TR_FRI (Inicio formulario) — El trámite no llega a iniciarse. Fallo previo al inicio del formulario (pasarela de autenticación Cl@ve / entorno de acceso).
Nota: Este es el caso más habitual en fallos generales de la pasarela Cl@ve. Si el error es de conexión/timeout y afecta a varios ciudadanos → posible fallo de pasarela Cl@ve.
✅ SÍ aparece TR_FRI (Inicio formulario) — (caso menos frecuente). El acceso se ha iniciado, pero falla inmediatamente después. Si el error es SAML de identidad (p. ej. Response is fail: Usuario no registrado presencialmente / Nivel de registro insuficiente / Usuario no autorizado) → posible error individual del ciudadano.
Si el error es de conexión/timeout y afecta a varios ciudadanos → posible fallo de pasarela Cl@ve (caso menos frecuente).
NO aparecen trazas en SistraHelp — Si NO hay ninguna traza = error previo al inicio del trámite. Posibles causas:
  • Si el error es de conexión/timeout y afecta a varios ciudadanos → posible fallo de pasarela Cl@ve.
  • Fallo general en la pasarela de autenticación Cl@ve / IdP.
  • Cl@ve no activa / registro Cl@ve no válido.
  • Certificado local no instalado, caducado o no confiable.
  • DNIe sin drivers, lector no detectado o PIN bloqueado.
  • Problema del navegador con certificados.

Mensajes típicos:

El navegador intenta redirigir a https://pasarela.clave.gob.es y la página queda en blanco tras pulsar «Iniciar trámite».
► No carga https://pasarela.clave.gob.es o https://pasarela.clave.gob.es/Proxy2/ServiceProvider.
Se puede probar acceder a Carpeta Ciudadana GOB o revisar el estado del servicio pasarela.
El acceso funciona de forma intermitente (a veces sí, a veces no).
No aparece selector de certificados / no solicita PIN.

Acción:

Si aparece TR_FRI y el ciudadano recibe error SAML Response is fail: Usuario no registrado presencialmente → Error del ciudadano (registro Cl@ve no presencial / no avanzado). CAI‑2626766.
Si NO hay trazas o el error es de conexión/timeout:
  • Indicar que el problema es previo al trámite.
  • Para confirmar posible fallo de pasarela Cl@ve, probar acceso a Carpeta Ciudadana CAIB o https://sede.dgt.gob.es/es/mi_dgt/.
  • Si tampoco cargan → se confirma fallo general de pasarela Cl@ve.
▶ Informar de posible incidencia general en la pasarela de autenticación Cl@ve o fallo del proveedor de identidad (IdP) si afecta a varios ciudadanos.
Recomendar reintentar más tarde.
No es un problema del trámite ni del equipo del ciudadano.

Error al firmar – Proveedor Cl@ve Firma

Escenario: Al intentar firmar el trámite, la firma falla y el sistema devuelve errores del proveedor Cl@ve Firma. Este problema puede ocurrir tanto accediendo con Cl@ve Permanente como con certificado digital local, si el trámite está configurado para firmar con Cl@ve Firma.

Trazas:

→ Aparece TR_SGI (Inicio firma)
→ Aparece TR_SGX (Firma KO)
❌ NO aparece TR_SGO (Fin firma OK)

Mensajes típicos:

Código Error: 8 – Tipo Resultado: 15
Código Error: 103 – (Certificados bloqueados)
Código Error: 101 (registro/credencial inconsistente)
Código Error: 104 (registro Cl@ve débil)
(VALIDATION) InvalidNotSignerCertificate: — Identifica un fallo del servicio de firma Cl@ve Firma. En las trazas aparece como literales de validación DSS/XSS propios del proveedor:
(VALIDATION) La validació del certificat del firmant contingut en el diccionari de firma [Signature1] no ha estat correcta
urn:oasis:names:tc:dss:1.0:resultmajor:Warning
urn:afirma:dss:1.0:profile:XSS:resultminor:InvalidNotSignerCertificate
Habitualmente acompañado de: Error general durant el procés de firma dels fitxers: 500

Qué significa:

El trámite firma con Cl@ve Firma aunque el ciudadano haya accedido con certificado local.
El proveedor de firma lo determina la configuración del trámite, no el método de acceso.

Acción:

Si afecta a varios ciudadanos → informar de problema general del servicio de firma Cl@ve.
Si afecta solo a un ciudadano → revisar credencial de firma Cl@ve (Error 8-15 / 103...).

Error Portafib / Error Formulario (Acceso‑Sesión)

Escenario: Después de pulsar «Continuar / Firmar», la firma no llega a iniciarse porque Portafib no puede generar la sesión de firma (error de flujo) o falla el formulario.

Trazas:

TR_FRF (Fin formulario) → OK — El ciudadano ha pulsado «Continuar / Firmar» y el formulario se ha validado correctamente.
❌ NO aparece TR_SGI (Inicio firma) — La firma no se inicia. Portafib falla antes de invocar al proveedor de firma.
⚠️ Si NO aparece TR_SGI = problema en la fase Acceso‑Sesión (previa a firma).

Mensajes típicos:

Fluxe no vàlid – Error de flujo.
Excepció sessió firma – Error de flujo.
SesionFirmaClienteException
No s'ha especificat presentador al script (obligatori si iniciador és anònim) – Error funcional de formulario.

Acción:

Si aparecen errores de flujo → ✉︎ Enviar mail «Problemas con el Servicio Portafib».
Si NO aparecen errores de flujo y el mensaje apunta a formulario → ✉︎ Enviar mail problema funcional del formulario. (Puede fallar formulario y no aparecer TR_FRF.)
Finalizar 603.

Pago no verificado

Escenario: El ciudadano no llega a iniciar la firma porque el pago del trámite no se valida correctamente.

Trazas SistraHelp:

(Pago no verificado)
TR_PAN – Pago no verificado

Interpretación: El trámite se detiene en la fase de pago. No se inicia la firma electrónica ni se invoca a ningún proveedor de firma.

Clasificación: Funcional (no firma)

Acción CAU: Indicar al ciudadano que revise el proceso de pago o contacte con el órgano gestor del trámite. ✉︎ Enviar mail «Problemas con el Pago».

Carga desde Carpeta

Escenario: El ciudadano intenta recuperar o continuar un trámite desde Carpeta Ciudadana y el proceso no avanza correctamente.

Trazas SistraHelp:

(Carga desde Carpeta)
TR_CAR – Carga del trámite

Interpretación: El problema se produce durante la recuperación del trámite. No está relacionado con la firma electrónica ni con los proveedores de firma.

Clasificación: Funcional (no firma)

Acción CAU: Indicar reintento desde Carpeta Ciudadana o inicio de un nuevo trámite si procede. No escalar como incidencia de firma.

Cl@ve (8–15)

Escenario: Cl@ve se abre y acepta las credenciales, pero la firma falla. El proveedor Cl@ve Firma devuelve el código 8–15.

Trazas:

TR_SGI (Inicio firma)
TR_SGX (Firma KO)

⚠️ Detección de posible fallo general:

Código 8–15 + varios NIF distintos + misma franja horaria → Posible incidencia general del proveedor Cl@ve Firma.

Acción:

Caso individual → renovar/acreditar Cl@ve Permanente. ✉︎ Enviar mail «Problema con Cl@ve Firma (8–15)».
Varios NIF → informar incidencia general y reintentar más tarde. ✉︎ Enviar mail «Problemas con el Servicio».

Nota CAU:

El 8–15 es el único código usado para detectar caída general.
101 y 103 son siempre individuales, salvo aviso oficial.

Cl@ve (103 – Bloqueo de Contraseña)

Escenario: El ciudadano llega a las pantallas de Cl@ve y efectúa más de cinco intentos de introducción incorrecta de la contraseña de Cl@ve Permanente.

Trazas:

TR_SGI (Inicio firma) — Portafib ha invocado correctamente a Cl@ve para iniciar la firma.
TR_SGX (Firma KO) — El proveedor Cl@ve Firma (nube) devuelve 103. La firma no se completa.

Mensajes típicos: Error Cl@ve Firma 103

Acción: Indicar que siga las instrucciones del apartado «olvido de contraseña». ✉︎ Enviar mail Cl@ve (credenciales).

🔹 Ver artículo completo: Error 103

Cl@ve (103 – Tipo 15 – Bloqueo Certificado)

Escenario: El ciudadano se autentica correctamente en Cl@ve, pero la firma falla porque su certificado Cl@ve está bloqueado. El proveedor Cl@ve Firma devuelve el código 103 – Tipo 15.

Trazas:

TR_SGI (Inicio firma) — Portafib invoca correctamente a Cl@ve para iniciar la firma.
TR_SGX (Firma KO) — El proveedor Cl@ve Firma (nube) devuelve 103 – Tipo 15. La firma no continúa.
⚠️ 103 – Tipo 15 indica certificados Cl@ve bloqueados / revocados.

Mensajes típicos: Certificados bloqueados (103 – Tipo 15)

Acción: Indicar al ciudadano que debe revocar y volver a emitir su certificado Cl@ve desde Gestiones Cl@ve. Recomendar revisar el nivel de registro Cl@ve (avanzado). ✉︎ Enviar mail Cl@ve 103‑15.

🔹 Ver artículo completo: Error 103

Cl@ve (101 – Registro / Nivel insuficiente)

Escenario: El ciudadano llega a las pantallas de Cl@ve, pero la firma no se completa porque no dispone de un certificado válido para firmar (registro Cl@ve incompleto, nivel insuficiente o certificado no emitido).

Trazas:

TR_SGI (Inicio firma) — Portafib ha invocado correctamente a Cl@ve para iniciar la firma.
TR_SGX (Firma KO) — El proveedor Cl@ve Firma (nube) devuelve 101. La firma no se completa.

Mensajes típicos:

Error Cl@ve Firma 101
No dispone de certificado para firmar
Nivel de registro insuficiente

Acción: Indicar al ciudadano que dispone de un registro Cl@ve débil y que debe realizar un registro Cl@ve avanzado mediante acreditación de su identidad. ✉︎ Enviar mail Cl@ve (Registro Cl@ve insuficiente).

🔹 Ver artículo completo: Error 101

Error Cl@ve Firma – Registro débil (Código 104)

Escenario: La firma se inicia correctamente (aparece TR_SGI), pero Cl@ve Firma rechaza la operación porque el ciudadano tiene un registro Cl@ve débil, que no permite disponer de certificados de firma.

Trazas:

TR_SGI (Inicio firma) → OK — Portafib invoca correctamente al proveedor Cl@ve Firma.
TR_SGX (Firma KO) — Cl@ve Firma devuelve error por nivel de registro insuficiente.
❌ NO aparece TR_SGO (Fin firma OK) — La firma no se completa.
❌ NO aparece TR_REG (Registro) — El trámite no llega a registrarse.

Mensajes típicos:

L'usuari no pot tenir certificats de signatura per haver realitzat un registre feble
Codi Error: 104 – Proveïdor: clavefirma – Tipus Resultat: 15

Interpretación:

El ciudadano tiene un registro Cl@ve débil.
Cl@ve Firma no permite emitir ni usar certificados de firma con este tipo de registro.
No es un fallo técnico ni una caída del servicio. Es un caso individual del ciudadano.

Acción:

Informar al ciudadano de que debe realizar un registro Cl@ve avanzado (acreditación fuerte).
Tras completar el registro avanzado, deberá reintentar la firma desde Carpeta Ciudadana.
No escalar como incidencia técnica.
✉︎ Enviar mail «Registro Cl@ve insuficiente (Código 104)».

Cl@ve Móvil – Método NO permitido

Escenario: Las pantallas de Cl@ve aparecen, pero el método elegido no es válido para firmar en CAIB (Cl@ve PIN, Cl@ve Móvil) o el ciudadano no tiene el nivel de registro/QAA necesario. La firma no llega a completarse.

Trazas:

TR_SGI (Inicio firma) — Portafib invoca a Cl@ve correctamente para iniciar la firma.
TR_SGX (Firma KO) — Cl@ve rechaza el método/credencial, pero sin códigos de proveedor (no 8–15, no 103).
⚠️ Si NO hay códigos 8–15 o 103, y SÍ hay pantallas Cl@ve → es un error de método / credencial.

Mensajes típicos: Métode CLAVE_MOVIL no permès

Acción:

Indicar al ciudadano que la firma SOLO funciona con Cl@ve Permanente.
Explicar que Cl@ve PIN y Cl@ve Móvil NO sirven para firmar en CAIB.
✉︎ Enviar mail Cl@ve Móvil Método No permitido.

Autofirma (certificado local / SAF_27 / SAF_xx)

Escenario: La firma llega a iniciarse (aparece TR_SGI), pero no se completa. El problema puede estar en el entorno local del ciudadano (Autofirma / certificado) o en el proveedor Autofirma en servidor.

Trazas:

TR_SGI (Inicio firma) — Portafib invoca correctamente al proveedor de firma.
TR_SGX (Firma KO)
Con SAF_27 → Posible problema en Autofirma Proveedor-Servidor.
Con otros SAF_xx y sin códigos Cl@ve → Posible problema del entorno local (certificado / cliente Autofirma).
Revisar detalle trazas TR_INI / TR_CAR: doble clic sobre la traza de inicio o carga del trámite. Si SO = IPHONE / IPAD → el trámite se ha iniciado desde dispositivo móvil.

Mensajes típicos:

SAF_27, SAF_xx
Mensajes del cliente Autofirma / Cliente de Firma Móvil.
Signatura cancel·lada — Significa que no se ha recibido una firma válida desde el proveedor. El sistema NO sabe si el ciudadano pulsó «Cancelar» o si hubo cancelación técnica (Autofirma falló, SSL cortado, plugin devolvió algo inválido, timeout...).
Autofirma no se abre / solicita escaneo de DNI / no detecta certificado.
El plugin amb ID null ha retornat un status OK per la petició de passarela - però el fitxer Signat està buit (CAI-2624660)

Acción:

Siempre pedir al ciudadano:
  • Realizar la firma desde un ordenador.
  • Desinstalar Autofirma, reiniciar equipo e instalar la aplicación Autofirma.
  • Comprobar que el certificado local está bien instalado y vigente y que Autofirma lo detecta.
  • Probar a firmar un PDF en local con Autofirma.
  • ✉︎ Enviar mail «Instalar / actualizar Autofirma y probar firma local».
En iPhone / iPad (opcional, sin garantía): Safari puede intentar abrir Autofirma como si fuera macOS (vista de escritorio). Acción: desactivar «Vista de escritorio» en Safari y reintentar. ✉︎ Enviar mail Autofirma iPhone / iPad.
En macOS: Problema habitual: Autofirma bloqueada por Gatekeeper. Regla práctica: si Autofirma no se abre manualmente, no se abrirá desde el navegador. Solución: permitir ejecución desde «Seguridad y privacidad», verificar que se abre manualmente, probar firma local de un PDF. ✉︎ Enviar mail Autofirma en macOS (aplicación bloqueada).
Si tras firmar desde ordenador sigue fallando:
  • Con SAF_27 → posible problema de Autofirma servidor. ✉︎ Enviar mail «Problemas con el Servicio».
  • Sin SAF_xx → problema del entorno local. Revisar tabla de asignación, apartado «Error al firmar con certificado / Otros errores al firmar».
Si hay varios NIF con el mismo error en la misma franja → posible incidencia general.

REGWEB – Justificante no generado / Fallo Registro

Escenario: La firma del trámite se completa correctamente, pero el justificante no se genera o el registro no finaliza. El error se produce después de TR_SGO, durante la fase de registro en REGWEB.

Trazas:

TR_SGO (Fin firma OK) — La firma se ha realizado correctamente.
TR_REG (Registro) — REGWEB devuelve error o no genera el justificante.
❌ Falta TR_FIN (Fin trámite) — El trámite queda sin cerrar.
⚠️ Firma correcta + registro KO = Error REGWEB (backend CAIB, no proveedor de firma).

Mensajes típicos:

El fluxe no es vàlid
AccionPasoNoPermitidaException
No se ha podido generar justificante
NIF no concorda
PDF corrupto
Justificante no generado
Registro incompleto

Acción:

Si aparece El fluxe no es vàlid / Registre està pendent de verificació (estat reintentar) / AccionPasoNoPermitidaException / Excepció al registrar. Marcam per reintentar / RegistroSolicitudException:
→ Suele ser un fallo puntual de REGWEB o causado por navegación no permitida (uso de botones atrás/adelante). No pasar CAI y pedir reintentar el registro.
Para el resto de casos: informar al ciudadano de que la firma es correcta, pero el justificante no se ha generado por un error de REGWEB. El trámite debe ser revisado por el equipo responsable del registro.
✉︎ Enviar mail «Problemas REGWEB».
Asignar a DGIDT REGWEB — APLICACIONS\7.-PENDENTS REGISTRE WEB
